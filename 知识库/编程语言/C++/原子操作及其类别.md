原子操作是一个或多个操作的序列，这些操作要么全部执行成功，要么完全不执行，并且执行过程不会被其他线程中断。
* 不可分割性
* 内存可见性
* 顺序保证
# 原子变量
原子变量是封装了原子操作的类型，提供对特定数据类型的线程安全访问。
C++20开始，所有平凡可复制类型都可以用于 `std::atomic<T>`，但是否无锁取决于硬件和编译器实现。
```C++
template<typename T>
constexpr bool is_suitable_for_atomic() {
    // 必要条件：可平凡复制
    if constexpr (!std::is_trivially_copyable_v<T>) {
        return false;
    }
    
    // 检查是否总是无锁（C++17特性）
    if constexpr (std::atomic<T>::is_always_lock_free) {
        return true;
    }
    
    // 其他情况：可能支持，但不保证无锁
    return true;  // 类型可用，但可能用锁实现
}
```
有锁的原子变量本质上是自动加锁的常规变量，而不是真正的硬件原子操作。使用时应该像使用任何其他需要锁保护的资源一样小心。

# CAS
**CAS（Compare-And-Swap）** 是一种**原子操作**，用于实现无锁（lock-free）编程。
CAS操作流程：
1. 读取内存位置的当前值
2. 比较当前值与预期值
3. 如果相等：
   a. 写入新值到内存位置
   b. 返回成功
4. 如果不相等：
   a. 将当前值写入expected引用
   b. 返回失败

	预期值通常是基于当前原子变量的值来设定的。常见的模式是：先读取原子变量的当前值，然后基于这个当前值计算出新值，最后使用CAS尝试原子地更新。如果在计算新值的过程中，原子变量的值已经被其他线程改变，那么CAS就会失败，这时需要重新读取当前值，再次计算新值，并重试CAS。
```C++
bool CAS(T* addr, T expected, T desired) {
    if (*addr == expected) {
        *addr = desired;
        return true;
    } else {
        return false;
    }
}
```
## 使用场景
1. 乐观锁
2. 无锁容器
3. 无锁计数器
4. 自旋锁
5. 引用计数
## ABA问题
ABA问题发生在以下场景：
	线程1读取共享变量的值为 A
	线程1被挂起（或发生上下文切换）
	线程2将变量从 A 改为 B，然后又改回 A
	线程1恢复执行，执行CAS操作，检查变量是否为 A（确实是A），然后成功将变量改为新值 C
表面上看起来CAS成功了，但实际上在这期间变量已经发生了变化！
ABA问题产生的根本原因是*CAS只检查值是否相等，不检查历史是否一致*。
### 无锁栈的例子
```C++
#include <atomic>
#include <memory>

template<typename T>
class LockFreeStack {
private:
    struct Node {
        T data;
        Node* next;
        Node(const T& d) : data(d), next(nullptr) {}
    };
    
    std::atomic<Node*> head{nullptr};
    
public:
    void push(const T& value) {
        Node* new_node = new Node(value);
        new_node->next = head.load();
        while (!head.compare_exchange_weak(new_node->next, new_node)) {
            // 重试
        }
    }
    
    std::unique_ptr<T> pop() {
        Node* old_head = head.load();
        while (old_head && 
               !head.compare_exchange_weak(old_head, old_head->next)) {
            // ABA问题可能在这里发生！
        }
        if (!old_head) return nullptr;
        auto result = std::make_unique<T>(old_head->data);
        delete old_head;  // 危险！可能删除错误的节点
        return result;
    }
};

// ABA问题场景：
// 1. 线程1：读取head = A，A->next = B
// 2. 线程1被挂起
// 3. 线程2：pop() 弹出A，delete A
// 4. 线程2：push() 新节点，恰好分配到相同内存地址A
// 5. 线程1恢复：CAS检查 head == A（确实相等！）
// 6. 线程1：设置head = B（但此时A已不是原来的A）
// 7. 结果：内存损坏或数据丢失

开始使用CAS操作
      ↓
需要处理指针吗？ → No → 直接使用CAS
      ↓ Yes
      ↓
数据会被释放并重用吗？ → No → 直接使用CAS
      ↓ Yes
      ↓
系统支持双字CAS吗？ → Yes → 使用版本号指针
      ↓ No
      ↓
性能要求极高吗？ → Yes → 使用危险指针
      ↓ No
      ↓
读操作远多于写吗？ → Yes → 使用RCU
      ↓ No
      ↓
使用引用计数 + 延迟回收
```

## 特点
### CAS的限制
1. ABA问题：需要额外机制解决
2. 只能更新单个内存位置：无法原子更新多个变量
3. 高竞争下的性能问题：可能导致大量重试
4. 平台依赖：不同硬件支持程度不同
