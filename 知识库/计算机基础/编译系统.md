# 目标文件
| 平台 / 操作系统          | 可执行文件格式                                 | 常见后缀                             | 主要特征                                                                               |
| ------------------ | --------------------------------------- | -------------------------------- | ---------------------------------------------------------------------------------- |
| **Windows**        | PE / PE32 / PE32+ (Portable Executable) | `.exe`, `.dll`, `.sys`, `.ocx`   | 基于 COFF，含 DOS 存根、PE 头、节表（`.text`, `.data`, `.rsrc`），支持导入/导出表，32/64 位分别称 PE32/PE32+ |
| **Linux / 类 UNIX** | ELF (Executable and Linkable Format)    | 无后缀（可执行），`.so`, `.o`, `.ko`      | 包含 ELF 头、程序头表（加载用）、节头表（链接用），支持多种 CPU 架构（x86, ARM, RISC-V 等）                        |
| **macOS / iOS**    | Mach-O (Mach Object)                    | 无统一后缀，`.dylib`, `.framework`     | 含 Mach 头、加载命令、多段（`__TEXT`, `__DATA`），支持通用二进制（Universal Binary，如 x86_64 + ARM64 合并） |
| **早期 UNIX**        | a.out                                   | `.out`                           | 结构简单，已被 ELF 取代，缺乏动态链接支持                                                            |
| **嵌入式 / 烧录**       | Intel HEX, SREC (S‑records), 纯二进制       | `.hex`, `.srec`, `.bin`          | 文本或二进制格式，直接对应存储器地址，用于微控制器、ROM 烧写                                                   |
| **跨平台中间格式**        | Java Class / JAR, WebAssembly, LLVM IR  | `.class`, `.jar`, `.wasm`, `.bc` | 不直接在 CPU 执行，需虚拟机（JVM, WASM 运行时）或 JIT 编译                                            |

| 类型 (e_type)               | 数值  | 常见后缀                            | 主要用途与特点                                                                                                    | 生命周期阶段                                                                                                                               |
| ------------------------- | --- | ------------------------------- | ---------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| **可重定位文件**  <br>`ET_REL`  | 1   | `.o`, `.a`                      | 编译器和汇编器生成的**中间产物**，包含代码和数据，但**地址未固定**，需要链接器处理才能成为可执行文件或共享库[](https://developer.aliyun.com/article/672017)。 | **编译/链接**时[](https://www.openeuler.org/zh/blog/lijiajie128/2020-11-03-ELF%E6%96%87%E4%BB%B6%E6%A0%BC%E5%BC%8F%E8%A7%A3%E6%9E%90)     |
| **可执行文件**  <br>`ET_EXEC`  | 2   | 无固定后缀 (如 `/bin/bash`)           | 包含完整代码和数据，指定了程序入口地址（`e_entry`），可被操作系统直接加载到内存中运行[](https://developer.aliyun.com/article/672017)。            | **运行**时[](https://www.openeuler.org/zh/blog/lijiajie128/2020-11-03-ELF%E6%96%87%E4%BB%B6%E6%A0%BC%E5%BC%8F%E8%A7%A3%E6%9E%90)        |

| **核心转储文件**  <br>`ET_CORE` | 4   | `core`, `core.pid`              | 进程意外崩溃（如段错误）时，系统转储的内存快照，用于事后**调试**（如使用 `gdb`），分析崩溃原因[](https://developer.aliyun.com/article/672017)。       | **异常/调试**时                                                                                                                           |
## 文件组织
ELF头（ELF Header）是 ELF 文件的第一个部分，**长度固定**（32 位环境下 52 字节，64 位环境下 64 字节）。
```C++
typedef struct {
    unsigned char e_ident[16];   // 魔数 + 标识信息
    uint16_t      e_type;        // 文件类型（可重定位、可执行、共享库等）
    uint16_t      e_machine;     // 目标架构（x86、ARM、RISC-V 等）
    uint32_t      e_version;     // 文件版本（通常为 1）
    uint64_t      e_entry;       // 程序入口点虚拟地址（可执行文件才有意义）
    uint64_t      e_phoff;       // 程序头表（Program Header Table）在文件中的偏移
    uint64_t      e_shoff;       // 节头表（Section Header Table）在文件中的偏移
    uint32_t      e_flags;       // 处理器特定标志
    uint16_t      e_ehsize;      // ELF 头本身的大小（64 字节）
    uint16_t      e_phentsize;   // 每个程序头表项的大小
    uint16_t      e_phnum;       // 程序头表项的数量
    uint16_t      e_shentsize;   // 每个节头表项的大小
    uint16_t      e_shnum;       // 节头表项的数量
    uint16_t      e_shstrndx;    // 节名字符串表（.shstrtab）在节头表中的索引
} Elf64_Ehdr;
```

## 链接
链接器负责把这些目标文件以及所需的库文件组合起来，生成一个**可执行文件**或**动态库**。
## 符号决议
链接器在将多个目标文件和库合并时，**将每个符号引用（Reference）与一个确切的符号定义（Definition）关联起来**的过程。
**特殊情况**
* 弱符号（Weak Symbol）：允许强定义覆盖弱定义，常用于库中的默认实现，用户可提供强定义覆盖。
* 公共块（Common Block）：Fortran 或旧的 C 中，未初始化的全局变量（int x;）在目标文件中是 SHN_COMMON 类型，链接器会合并多个同名公共块，取最大大小。
* 动态链接：共享库中的符号决议推迟到运行时（通过 LD_LIBRARY_PATH、DT_NEEDED 等）。

## 地址和空间分配
为每个输入目标文件的节（Section）分配最终的虚拟地址（VMA）和运行时存储空间。主要包括两个层面：
1. **虚拟地址空间中的位置**（链接视图 → 执行视图的段）
2. **输出文件中的物理偏移**（可执行文件本身的布局）

链接器为目标文件分配地址和空间的过程可以概括为：
1 . **读取所有输入目标文件**，收集各节的属性、大小和对齐要求。
2. **根据链接脚本或默认规则**，将输入节合并成输出节，并为每个输出节分配：
    - **虚拟地址（VMA）**：运行时位置。
    - **文件偏移**：对于 `PROGBITS` 节，在输出文件中连续放置；`NOBITS` 节不占空间。
3. **考虑对齐**，确保每个输出节的起始地址满足对齐要求。
4. **构建程序头表**，描述内存段的加载方式（VMA、文件偏移、权限、是否清零等）。 
5. **执行重定位**，根据最终分配的地址修正所有符号引用。

### 动态链接
#### 地址无关代码（PIC， Position-independent Code）
* 模块内部的函数调用和数据访问使用相对地址
* 模块间的函数调用和数据访问使用**全局偏移表（GOT）**，全局偏移表：代码地址无关，把和地址相关的部分放到数据段。
* 定义在模块内部的全局变量使用全局偏移表，避免产生多个副本
#### 装载
**可执行文件有一个明确的、唯一的入口点（Entry Point），而动态加载库（共享库）没有这样的入口点，取而代之的是初始化和终止函数机制**。

| 方式                           | 装载时机                                  | 触发者                                           | 典型场景                             | 优点                             | 缺点                  |
| ---------------------------- | ------------------------------------- | --------------------------------------------- | -------------------------------- | ------------------------------ | ------------------- |
| **启动时装载**（Eager / Load-time） | 进程初始化阶段，在 `main()` 执行之前               | 动态链接器（如 `ld-linux.so`）自动加载 `DT_NEEDED` 指定的共享库 | 普通动态链接的可执行文件（如 Linux 下的 `a.out`） | 程序运行期间不会因缺少库而失败；所有符号在启动时已确认存在  | 启动时间较长；可能加载了实际未使用的库 |
| **运行时按需装载**（Lazy / Runtime）  | 程序显式调用 `dlopen()` / `LoadLibrary()` 时 | 程序自身通过 API 主动加载                               | 插件架构、延迟加载大模块、可选功能                | 减少启动时间；节省内存（未使用的库不加载）；支持动态发现模块 | 需要额外编程；可能运行时因库缺失而失败 |
|                              |                                       |                                               |                                  |                                |                     |

| 方式                        | 重定位时机             | 实现机制                                               | 典型标志                                     | 优点                           | 缺点                                     |
| ------------------------- | ----------------- | -------------------------------------------------- | ---------------------------------------- | ---------------------------- | -------------------------------------- |
| **立即重定位**（Now / Bind now） | 库加载时立即完成所有符号地址解析  | 动态链接器遍历重定位表（`.rela.dyn`、`.rela.plt`），一次性填入 GOT/PLT | `LD_BIND_NOW=1` 或 `RTLD_NOW`（`dlopen` 时） | 运行时无额外开销；调用函数不会因重定位而暂停；行为可预测 | 加载时间更长（尤其库多时）；可能解析了实际未调用的函数            |
| **延迟重定位**（Lazy / PLT）     | 第一次调用函数时才解析该函数的地址 | 通过 PLT 桩代码 + 动态链接器介入，首次调用后 GOT 被修正为真实地址            | 默认行为（`RTLD_LAZY`）                        | 启动更快（只解析用到的函数）；减少不必要的重定位工作   | 第一次调用有微小延迟；可能因符号缺失在运行时才崩溃（不如立即绑定早暴露错误） |
